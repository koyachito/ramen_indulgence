# Issue 作業フロー

GitHub Issue に着手するときは、次の手順を守る。

1. 現在のブランチと作業ツリーを確認する。
   - `git branch --show-current`
   - `git status --short`
2. 未コミットの変更がある場合は、新しい Issue の作業を開始しない。
   - 既存の変更を勝手に破棄、上書き、commit、stash しない。
   - 変更の扱いが不明な場合はユーザーに確認する。
3. 前の Issue のブランチから `main` に移動する。
   - `git switch main`
4. 前の Issue が `main` にマージ済みであることを確認し、前の Issue のブランチをローカルと GitHub から削除する。
   - マージ確認: `git merge-base --is-ancestor <前のブランチ名> main`
   - ローカル削除: `git branch -d <前のブランチ名>`
   - GitHub 削除: `git push origin --delete <前のブランチ名>`
   - マージされていないブランチを強制削除しない。
5. GitHub の最新版を取り込む。
   - `git pull --ff-only origin main`
6. Issue 専用の新しいブランチを作成して移動する。
   - ブランチ名は `issue-<番号>-<短い説明>` とする。
   - 例: `git switch -c issue-10-mobile-about-link`
7. Issue の要件を確認し、必要な変更だけを実装する。
8. Issue の完了条件に対応するテストと動作確認を行う。
9. ローカル確認用サーバーを起動し、ユーザーがブラウザで確認できる状態にする。
   - 原則として `./start.sh` を使用する。
   - 起動後に `http://127.0.0.1:8000/` へアクセスできることを確認する。
   - ユーザーへ確認URLを報告する。
10. 差分を確認し、Issue に関係するファイルだけを commit する。
11. 作業ブランチを GitHub に push する。
   - 初回: `git push -u origin <ブランチ名>`
   - 以降: `git push`
12. 実装内容、確認結果、push 先のブランチをユーザーへ報告する。
   - Issue 本文の「やること」と「確認」の各チェック項目を最終的に照合する。
   - 各項目が満たされているかを報告し、未達項目がある場合は明記する。
   - GitHub へ push した場合は、必ず Pull Request 作成用 URL も報告する。
   - URL 形式: `https://github.com/<owner>/<repository>/pull/new/<ブランチ名>`

## Git 操作上の注意

- `main` 上で直接実装しない。
- `git pull` は意図しないマージコミットを避けるため `--ff-only` を使う。
- ユーザーの既存変更や未追跡ファイルを、Issue の変更へ混ぜない。
- `git reset --hard`、`git checkout -- <file>`、無断の `git stash` は行わない。
- push 前にテスト結果と `git diff`、`git status` を確認する。

## 確認・調査

- 状態を破壊しない確認、調査、テスト、差分閲覧、ローカル動作確認は、都度ユーザーへ許可を求めず、必要なだけ実行してよい。
- 確認用ローカルサーバーの起動、再起動、停止は、都度ユーザーへ許可を求めず実行してよい。
- 既存ファイル、既存データベース、ブランチ、GitHub などの外部状態を書き換える操作は、非破壊の確認には含めない。

## Push 前チェックリスト

push および PR 作成前に、以下を必ず実施すること。

### 1. pytest（全テスト）

```bash
source .venv/bin/activate && python -m pytest tests/ -q
```

全テストが pass であること。

### 2. Playwright E2E（Chromium + mobile）

```bash
npx playwright test --reporter=line
```

- `chromium` プロジェクト: Chromium ブラウザ
- `mobile` プロジェクト: Pixel 5 エミュレーション（Chromium ベース）

全テストが pass であること。

### 3. 7 ペルソナ QA

テストコードによる自動チェック (`tests/test_qa_personas.py`) で 39 テストが pass であること。

各ペルソナの観点は以下の通り。

| ペルソナ | 役割 | 主な観点 |
|----------|------|----------|
| P1 Newbie | 初めてのユーザー | トップページ→診断→結果の基本フロー完走 |
| P2 Veteran | ヘビーユーザー | reroll・全結果タイプ・全ラーメンタイプの出力確認 |
| P3 Malicious | 悪意ある攻撃者 | 不正な入力値・SQLインジェクション・XSS試行を送信 |
| P4 Data Integrity | データ品質担当 | レート制限・重複記録なし・統計カウント整合性 |
| P5 Migration | マイグレーション担当 | SQLite/PostgreSQL 切り替え時の動作確認 |
| P6 Regression | リグレッション担当 | 既存機能が壊れていないことの確認 |
| P7 Spec Skeptic | 仕様懐疑派 | エッジケース・境界値・仕様の穴を突く |

手動でブラウザ確認する際は、各ペルソナの観点でゴールデンパスとエッジケースを試すこと。

### 4. ブラウザ実動作確認

ローカルサーバー (`http://127.0.0.1:8000/`) を起動し、以下を目視確認する。

- トップページが表示される
- 診断を 1 件完走し、結果ページが表示される
- `/stats` で統計が表示される（グラフバーが正しく描画される）
- reroll ボタンで別文言の結果が表示される
- スマホサイズ（幅 390px 前後）で崩れがないこと

### 5. スマホサイズ挙動確認

ブラウザの DevTools または Playwright の mobile プロジェクトで 390×844 の viewport を確認すること。

## セキュリティレビュー手順

大きな変更後や定期的に、以下の観点でコードを調査し `findings.md` にまとめること。

### Phase 1: 調査のみ（コード変更なし）

1. 全ファイルを読む（`app/`・`tests/`・設定ファイル）
2. 以下の観点で問題を洗い出す:
   - **認証・認可**: セッション・Cookie・レート制限の実装
   - **インジェクション**: SQL・XSS・テンプレートインジェクション
   - **秘密情報**: ハードコードされたシークレット・環境変数の扱い
   - **ヘッダー**: セキュリティヘッダー（CSP・HSTS・X-Frame-Options 等）
   - **依存関係**: 脆弱なバージョン・不要な依存
   - **コード品質**: 未使用エクスポート・命名の不整合・パフォーマンス
   - **デプロイ**: Dockerfile・CI/CD・環境変数の漏洩リスク
3. 各指摘を `findings.md` に以下のフォーマットで記録する:

```
### F001 — タイトル

| 項目 | 内容 |
|------|------|
| **深刻度** | CRITICAL / HIGH / MEDIUM / LOW / INFO |
| **ファイル** | `path/to/file.py:行番号` |

説明・影響・修正方針
```

4. ユーザーに提示し、承認を待つ。**コードは変更しない。**

### Phase 2: 承認済み項目のみ修正

ユーザーが承認した ID のみを修正し、pytest + Playwright を実行してから push する。

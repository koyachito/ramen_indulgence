# AIと一緒にアプリを作って学んだこと

このドキュメントは「ラーメン免罪符」開発を通じて、AIとの協働から得た知見をまとめたものである。  
ChatGPT による仕様壁打ち → Codex による実装補助 → Claude Code によるイシュー実装・セキュリティレビューという流れで積み上がった学びを記録する。

---

## 1. 使ったAIと役割の変化

| フェーズ | AI | 主な使い方 |
|----------|-----|-----------|
| 企画・仕様壁打ち | ChatGPT | 案の言語化・比較。採用判断は自分 |
| 実装補助（Ver0.1→1.0） | Codex | 既存コードへの変更・テスト更新 |
| 残りイシュー一括実装 | Claude Code | ブランチ管理・PR作成・E2Eテスト |
| セキュリティレビュー | Claude Code | 調査→承認→修正の2段階フロー |

それぞれのAIは「丸投げ先」ではなく、役割が異なる補助者として機能した。

---

## 2. ChatGPTでの仕様壁打ちから学んだこと

### 良かった点

- **一人では固定されやすいアイデアが広がった。** 最初に思いついた案を唯一の正解にせず、複数案を比較できた。
- **「成し遂げたこと」の選択肢拡張**（推し活・人にやさしくした・何もしていない等）は、壁打ちで出てきた案を取捨選択した結果である。

### 落とし穴

- **案が増えるほど仕様が膨らむ。** AI は「できること」を出し続けるが、スコープを絞るのは人間の仕事。
- **採用した案がUXとして正しいとは限らない。** Cookie 判定や画像自動添付は実装してみて初めて問題が明確になった。

### 次に活かすこと

- 壁打ちセッション中に「これはA優先度か、C優先度か」を並行して決める。
- 「実装できるか」と「採用するか」は別の問いとして意識的に分ける。

---

## 3. Codexでの実装補助から学んだこと

### 良かった点

- **既存コードを前提とした修正**ができた。一括生成ではなく、Ver0.1の構造を維持しながら段階的に変更した。
- **ドキュメントを仕様として渡す**ことで、チャットだけでは失われる前提が共有できた（README・feedback.md・テストコードを入力として使用）。

### 落とし穴

- **最新仕様が会話に分散しやすい。** 仕様変更のたびに複数ファイルをContext として渡し直す必要があり、古いコンテキストで作業されるリスクがあった。
- **テストが仕様変更に追従する負担が大きかった。** 変更のたびにテストも書き直す循環が発生した。

### 次に活かすこと

- 仕様変更の理由をIssueやADRへ残す（採用した理由だけでなく、撤回した理由も）。
- 変更しやすい文言テストと、変わらない不変条件のテストを分離する。

## 4. Claude Codeでの開発から学んだこと（今日の作業）

### 4.1 イシュー一括実装とQA

**やったこと：** GitHub Issue #36〜#44（8件）を1ブランチで一気に実装。  
セキュリティヘッダー、Dependabot、E2Eテスト（Playwright）、QAペルソナテストを追加。

**学んだこと：**

- **7ペルソナQA**という枠組みが有効だった。コードレビューとは別に「壊しにいく視点」を明示することで、テストの網羅性が上がった。

  | ペルソナ | 観点 |
  |----------|------|
  | P1 Newbie | 基本フローが完走できるか |
  | P2 Veteran | 全結果タイプ・全パターンが出るか |
  | P3 Malicious | 不正入力・インジェクションを弾けるか |
  | P4 Data Integrity | レート制限・統計の整合性 |
  | P5 Migration | SQLite/PostgreSQL 切り替え時の動作 |
  | P6 Regression | 既存機能が壊れていないか |
  | P7 Spec Skeptic | 境界値・仕様の穴 |

- **push前チェックリストをコードベースに書いておく（AGENTS.md）と機能する。** 口頭で「テストしてから」と言うより、ファイルに書いてある方がAIも人間も一貫して実行できる。

- **Playwright E2Eはpytestで取れない「実際にブラウザで動くか」を担保する。** HTTPレベルのテストだけでは見えない、JavaScriptの動作・アニメーション完了後の状態を確認できる。

### 4.2 セキュリティレビュー（2段階フロー）

**やったこと：** Phase 1（調査のみ・findings.md作成・承認待ち） → Phase 2（承認済みのみ修正）

**学んだこと：**

- **「調査と修正を分ける」という指示は明示しないとAIは一緒にやろうとする。** AGENTS.mdやプロンプトに「ここで止まれ」と書くことで意図した通りに動く。

- **実際に見つかった問題と修正の対応関係：**

  | ID | 問題のパターン | 対応方針 |
  |----|---------------|----------|
  | F001（MEDIUM） | クライアント側で改ざんできる値をサーバーが信頼していた | HMAC署名で改ざん検知 |
  | F002（LOW） | .dockerignoreがなく意図しないファイルがイメージに混入しうる | 除外リスト作成 |
  | F003（LOW） | セキュリティヘッダーの抜け（HSTS） | ヘッダー追加 |
  | F004（LOW） | CSP の unsafe-inline が残っていた | JS経由のスタイル設定に変更 |
  | F005（LOW） | 本番向けでない起動フラグ（--reload） | フラグ削除 |
  | F006-F010（INFO） | 命名・型・キャッシュ・タイムゾーンの不整合 | 個別修正 |

- **セキュリティ問題は「攻撃できるか」だけでなく「意図と実装がずれていないか」の確認でもある。** 今回のレート制限は機能はしていたが、クライアントが自由に値を書き換えられる設計になっていた。

- **INFO レベルの問題でも、放置すると将来の混乱源になる。** 命名（プライベート関数が公開APIにいる）やデフォルト値（使われないNone）は、コードが大きくなると読み解くコストが上がる。

### 4.3 AIとのワークフロー管理

**学んだこと：**

- **AGENTS.mdはAIへの「ルールブック」として機能する。** ブランチ命名規則・テスト要件・push前チェックをリポジトリに置くことで、会話を超えて引き継がれる。

- **コンテキストが切れてもAGENTS.mdがあれば再開できる。** 今回のセキュリティレビューはセッションをまたいで続いたが、AGENTS.mdとfindings.mdがあったため状態を共有できた。

- **「承認欄つきのfindings.md」は、AIと人間の合意形成に使える。** 修正前に一覧を見せ、どれを直すか決める形式にすることで、AIが勝手に進める問題を防げる。

## 5. Codex → Claude Codeで変わったこと

| 観点 | Codex | Claude Code |
|------|-------|------------|
| コンテキストの与え方 | ファイルを手動で貼る | リポジトリを直接参照 |
| ブランチ・PR管理 | 自分で行う | AIが実行（git/gh コマンド） |
| テスト実行 | 自分で実行して結果を貼る | AIが実行して結果を確認 |
| 多ステップタスク | 1指示1作業 | 複数ステップを計画して実行 |
| エラー対応 | 自分がエラーを見てAIに貼る | AIがエラーを見て自己修正 |

Claude Code では「AIに状況を説明する」コストが大幅に下がった。ただし、**判断の主体が人間であることは変わらない。** 承認・撤回・方針決定は常に人間が持つ。

## 6. 「AIを使ってアプリを作る」上での普遍的な学び

### 6.1 AIは「思考の外部化」に使う

- 頭の中にある曖昧な案をAIへ出すことで、比較・検討できる形になる。
- 「AIが出した案」ではなく「比較した上で選んだ案」が自分のものになる。

### 6.2 実際に動かしてから判断する

- Cookie 判定・画像共有・隠し結果——どれも設計段階では「良さそう」だったが、動かしてみて変えた。
- AIの出力は「仮説」として扱い、プロトタイプして検証する。

### 6.3 ドキュメントが仕様書になる

- README・feedback.md・AGENTS.md・findings.md——これらはAIへの入力として機能する。
- 「後から読んでも分かる」ドキュメントは、AIとの協働でも「前提が壊れない」文脈を作る。

### 6.4 AIに「止まれ」と言える設計にする

- 多段階タスクでは、途中に確認ポイントを明示的に置く。
- 「調査だけして止まれ」「承認済みだけ直せ」のように、AIの行動範囲を文章で定義できる。

### 6.5 スコープは人間が守る

- AIは「できること」を増やし続けるので、仕様が膨らむ方向へ引っ張られる。
- 「今回はここまで」を決めるのは常に人間。優先度A・B・Cの区分は実装前から持っておく。

---

## 7. 次のプロジェクトに持ち越すこと

- [ ] **優先度A/B/Cを先に定義する。** 実装中に「ついでにやろう」を減らす。
- [ ] **受け入れ条件を短く固定する。** 何ができれば完了か、何は対象外かを最初に書く。
- [ ] **仕様変更の理由をIssueへ残す。** 採用した理由と同じくらい、撤回した理由が後から重要になる。
- [ ] **セキュリティレビューを定期的に行う。** 機能追加のたびに findings.md を更新するサイクルを作る。
- [ ] **7ペルソナQAをpush前チェックに組み込む。** テストコードだけでなく、手動ブラウザ確認も含めたチェックリストとして AGENTS.md に置く。
- [ ] **ユーザー検証を定量化する。** 次回は「どの質問で離脱したか」「画像保存を使ったか」など、行動を記録する仕組みを検討する。

# 開発作業フローのトラブルと反省点

作成日: 2026-06-25

この文書は、Issue対応とGit運用で実際に起きたトラブル、および既存の作業指示から読み取れる再発リスクを整理し、次回以降の行動へつなげるための記録である。

## これまでにあったトラブル

- Issue #14では、最初の実装をpush・mergeした後に実機で表示上の問題を発見し、同じブランチから追加修正とmergeを繰り返した。
  - PR #29: コンセプトコピーの追加
  - PR #30: 文字サイズと位置の修正
  - PR #31: 大きな画面での文字サイズ調整
- Issue #14の見た目変更について、commit前のローカル確認が不足していた。
  - ブラウザの開発者ツールでスマートフォン幅や大きな画面幅を確認していなかった。
  - そのため、実機確認が実装確認ではなく不具合発見の最初の機会になった。
- 過去のIssue作業で、ブランチ作成前に`main`へ移動して最新版をpullする手順を忘れ、古い`main`を起点に作業してコンフリクトが発生した。
- 前のIssueで使用したブランチが、merge後もローカルまたはGitHubに残ることがあった。
  - 作業開始時に現在のブランチを明示的に確認しないと、前のIssueのブランチを起点に次の作業を始める危険があった。
- Issue #10の機能修正ブランチに、ブランチ整理やPR URL報告に関する運用ドキュメントの変更も含まれた。
  - 機能変更と作業フロー改善が同じブランチへ混ざり、Issueの差分範囲が分かりにくくなった。
- push後に、push先ブランチやPull Request作成用URLの報告が不足したことがあった。
- 個人的なメモや未追跡ファイルをIssueの変更へ混ぜる危険があり、後から除外設定が必要になった。
- 自動テストや構文検査だけでは、文字サイズ、配置、折り返し、画面幅ごとの見え方などの視覚的不具合を検出できなかった。

## 次へ活かす反省点

- UI変更は、実装直後ではなくcommit前にローカルで目視確認する。
- ブラウザの開発者ツールを使い、少なくとも次の表示幅を確認する。
  - 小さいスマートフォン幅
  - 一般的なスマートフォン幅
  - タブレット相当の幅
  - PCの標準的な幅
  - 大きなPC画面
- レスポンシブ表示では、文字サイズだけでなく、位置、折り返し、重なり、余白、ナビゲーションの欠落も確認する。
- 実機確認は必要だが、ローカル確認を省略する理由にはしない。
  - ローカル確認で基本的な不具合を除去する。
  - 実機確認では端末固有の差や操作感を確認する。
- commit前の完了条件を「テストが通る」だけにせず、「対象画面を必要な表示幅で確認済み」まで含める。
- 新しい作業を始める前に、必ず現在のブランチと作業ツリーを確認する。
  - `git branch --show-current`
  - `git status --short`
- 未コミットの変更や未追跡ファイルがある場合は、新しいIssueの作業を始めない。
- 新しいブランチは、必ず最新の`main`から作成する。
  - `git switch main`
  - `git pull --ff-only origin main`
  - `git switch -c <作業ブランチ>`
- 前のIssueのブランチは、`main`へのmergeを確認してからローカルとGitHubの両方で削除する。
- 機能変更、運用ルール変更、個人的なメモを同じcommitやブランチへ混ぜない。
- `git add -A`へ安易に頼らず、Issueに関係するファイルだけを明示的にstageする。
- push前に、テスト結果、`git diff`、`git status`を確認する。
- push後は、実装内容、確認結果、push先ブランチ、Pull Request作成用URLをまとめて報告する。
- merge後の追加修正を完全になくすことは目標にしない。ただし、開発者ツールで事前に見つけられる問題をmerge後まで持ち越さない。

## commit前チェックリスト

- [ ] 最新の`main`から作業ブランチを作成した
- [ ] Issueと無関係な変更や未追跡ファイルが混ざっていない
- [ ] 対象の自動テストと構文検査が通っている
- [ ] `git diff --check`が通っている
- [ ] UI変更を複数の表示幅で目視確認した
- [ ] 必要に応じて実機でも確認した
- [ ] `git diff`で意図した変更だけであることを確認した
- [ ] `git status`でstage対象を確認した

## push後チェックリスト

- [ ] 正しい作業ブランチをpushした
- [ ] CI結果を確認した
- [ ] Pull Request作成用URLを共有した
- [ ] merge後に作業ブランチを削除した

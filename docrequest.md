# Codexへの依頼

このリポジトリについて、ポートフォリオ用の独立したドキュメントを2つ作成してください。

目的は、単に実装内容を書くことではなく、
「Ver0.1からVer1.0に向けて、仕様がどのように変化したか」
「AIをどのように活用しながら、最終的な仕様判断を人間が行ったか」
を第三者に伝わる形で記録することです。

以下の方針で作成してください。

---

# 1. 作成してほしいドキュメント

## 1つ目：仕様変化のドキュメント

ファイル名案：

```md
docs/spec_evolution_v0.1_to_v1.0.md
```

目的：

README.mdに書かれている昨日までの実装、つまり Ver0.1 の仕様と、
feedback.md および今日のオーダーに基づく Ver1.0 ブランチでの仕様変更案を比較し、
このアプリの仕様がどう変化したかを整理してください。

このドキュメントは、README.mdの置き換えではなく、独立した設計変更記録として作ってください。

書いてほしい内容：

* Ver0.1 の概要
* Ver0.1 で実装済みだった主要機能
* Ver0.1 の課題
* 友人からのフィードバックで明らかになった課題
* Ver1.0 で追加・変更する仕様
* 仕様変更の意図
* 優先度ごとの整理
* Ver0.1 → Ver1.0 の変更表
* 今後の拡張余地

特に、以下の観点を含めてください。

### Ver0.1 側

README.mdの内容を確認し、Ver0.1時点の仕様を整理してください。

例：

* FastAPI + Jinja2 による診断アプリ
* ラーメンを食べる理由を入力から生成する
* 診断結果画面を表示する
* Renderにデプロイ済み
* 画像共有や投稿導線は限定的
* 統計や永続化は未実装、または限定的
* 質問数や分岐はまだ少なめ
* 結果文言はやや固定的

README.mdに実際に書いてある内容を優先し、推測で断定しないでください。

### Ver1.0 側

今日のオーダー内容と feedback.md を踏まえ、Ver1.0で目指す仕様を整理してください。

### 変更表の例

以下のような表を入れてください。

| 項目   | Ver0.1   | Ver1.0案          | 変更意図            |
| ---- | -------- | ---------------- | --------------- |
| 質問数  | 少なめ      | 懺悔感を出すため増加       | 体験の厚みを出す        |
| 結果文言 | 固定的      | 徳目・免罪対象・締め文を追加   | 共有したくなる情報量にする   |
| 共有導線 | 限定的      | 画像DL・投稿文コピー・Xを開く | SNS共有しやすくする     |
| 隠し結果 | 隠しコマンド中心 | 通常診断からも到達可能に     | ネタ要素を見てもらいやすくする |
| 統計   | 限定的      | PostgreSQLで保存検討  | 継続利用・ポートフォリオ性向上 |

## 2つ目：AI活用振り返りのドキュメント

ファイル名案：

```md
docs/ai_development_retrospective.md
```

目的：

このプロジェクトでのAI活用方法について、ポートフォリオで読める形に整理してください。

単に「AIを使いました」と書くのではなく、
「どこをAIに相談し、どこを自分で判断し、どこをユーザー検証で修正したか」
がわかるドキュメントにしてください。

書いてほしい内容：

* 開発全体におけるAIの役割
* ChatGPTを使った仕様壁打ち
* Codexを使った実装支援
* 最終判断は自分で行ったこと
* Ver0.1をデプロイしたこと
* 友人9人からフィードバックを受けたこと
* その内容を feedback.md に整理したこと
* フィードバックをもとに Ver1.0 ブランチで改善仕様を切ったこと
* AI活用の良かった点
* AI活用の危うい点
* 今回の使い方がポートフォリオとしてどう見えるか
* 次回以降改善したい点

### 必ず含めてほしい観点

このプロジェクトでは、AIは「仕様を丸投げして完成品を作らせる道具」ではなく、以下のように使ったことを整理してください。

* ChatGPT

  * アイデア整理
  * 仕様の壁打ち
  * 文言案の比較
  * 体験設計の相談
  * ランダム要素や共有導線の是非の検討
  * ユーザーにどう見えるかの整理
* Codex

  * 実装作業の補助
  * 既存コードの修正
  * READMEやfeedback.mdなど、リポジトリ内の実ファイルを前提にした作業
  * 変更内容のドキュメント化
* 自分

  * 何を作るかを決めた
  * どの案を採用するかを決めた
  * Ver0.1を実際にデプロイした
  * 友人9人に触ってもらった
  * フィードバックを受け取った
  * feedback.mdに整理した
  * Ver1.0で何を変えるか判断した

### 書きぶりの注意

以下のようなニュアンスにしてください。

* AIに作らせただけ、という印象にしない
* かといって、AIを使っていないように見せる必要もない
* AIを「思考の外部化」「実装補助」「レビュー相手」として使った、と整理する
* 最終的な意思決定者は自分であることを明確にする
* 友人9人からのフィードバックを受けて改善している点を、ユーザー検証として評価する
* AI時代の開発スタイルとして、現実的で再現性のある使い方だったかをCodexの視点で振り返る

### 章立て案

以下のような構成にしてください。

```md
# AI活用振り返り

## 1. このドキュメントの目的

## 2. プロジェクト概要

## 3. AIを使った範囲

## 4. AIに任せなかった範囲

## 5. ChatGPTとの仕様壁打ち

## 6. Codexによる実装支援

## 7. Ver0.1のデプロイとユーザーフィードバック

## 8. feedback.mdからVer1.0仕様への反映

## 9. このAI活用方法の評価

## 10. 課題と改善点

## 11. ポートフォリオとして伝えたいこと
```

### 評価の観点

Codex自身の判断として、今回のAI活用方法を以下の観点で振り返ってください。

* 仕様を自分で考えていたか
* AIに丸投げしていなかったか
* ユーザーの反応を取り入れていたか
* 小さくデプロイしてから改善していたか
* 実装と仕様の往復ができていたか
* ポートフォリオとして説明可能なプロセスになっているか
* 改善すべき点は何か

特に、以下のような評価を入れてください。

* Ver0.1を先にデプロイした点は良い
* 友人9人からフィードバックを取った点は良い
* feedback.mdに整理した点は良い
* ChatGPTとの壁打ちで案を増やしつつ、最終判断を自分で行った点は良い
* 一方で、仕様が膨らみやすいので、優先度A/B/Cに分ける必要がある
* AIが出した案をそのまま採用せず、実装コスト・ユーザー体験・ポートフォリオでの見え方を踏まえて取捨選択する必要がある

# 2. 作業時の注意

* README.md、feedback.md、現在のブランチ上のファイルを確認してから書いてください
* 推測だけで断定しないでください
* 実際にリポジトリ内に存在する内容を優先してください
* 今日のオーダー内容は Ver1.0 の改善案として扱ってください
* まだ実装されていない内容は「実装済み」ではなく「Ver1.0案」「改善予定」「設計方針」として書いてください
* ポートフォリオで採用担当や開発者が読むことを想定し、ふざけすぎず、読みやすい文体にしてください
* ただし、ラーメン免罪符というアプリのユーモアは消さないでください

# 3. 最終的にやってほしいこと

3. 今日のVer1.0向けオーダー内容を踏まえる

6. 作成後、変更ファイル一覧を示す
7. 可能なら、各ドキュメントの要約を短く出す
